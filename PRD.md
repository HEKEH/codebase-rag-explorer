# 📘 项目需求文档（精简执行版）

## 项目名称

Codebase RAG Explorer（代码库智能问答系统）

---

## 一、项目目标

构建一个系统，使用户能够对代码库进行自然语言提问，系统通过检索相关代码并生成回答，帮助用户理解代码结构与逻辑。

---

## 二、范围定义

## 2.1 必须实现（MVP）

系统需支持以下完整流程：

1. 导入代码库
2. 对代码进行结构化切分
3. 建立可检索索引
4. 接收用户问题
5. 检索相关代码片段
6. 基于检索结果生成回答
7. 返回回答及对应代码引用

---

## 2.2 不在本期范围

以下能力明确不实现：

- 自动修改或生成代码文件
- 多步骤任务执行（Agent 行为）
- 权限控制与用户系统
- 实时代码同步更新
- 多仓库联合查询

---

## 三、核心用户流程

## 流程1：代码库问答

步骤：

1. 用户导入一个代码库
2. 用户触发索引构建
3. 用户输入自然语言问题
4. 系统返回：
   - 回答内容
   - 相关代码片段
   - 代码所在路径

---

## 四、功能需求（Functional Requirements）

## 4.1 代码库导入

### 4.1 输入

- 本地目录路径 或 仓库地址

### 4.1 行为要求

- 系统读取所有源代码文件
- 系统忽略非代码文件及指定目录（如依赖目录、构建目录）

### 4.1 输出

标准化文件数据结构：

```json
{
  "path": "string",
  "content": "string"
}
```

---

## 4.2 代码切分

### 4.2 目标

将代码拆分为“语义完整单元”，用于后续检索

### 4.2 行为要求

- 优先按以下结构切分：
  - 函数
  - 类

- 若无法识别结构，则退化为固定长度切分

### 4.2 输出结构

```json
{
  "chunk_id": "string",
  "content": "string",
  "file_path": "string",
  "metadata": {
    "type": "function | class | generic",
    "name": "string | null"
  }
}
```

---

## 4.3 索引构建

### 4.3 行为要求

- 系统为每个 chunk 构建可检索索引
- 每个 chunk 必须能被唯一识别并检索

---

## 4.4 问题处理

### 4.4 输入

用户自然语言问题：

```json
{
  "question": "string"
}
```

---

## 4.5 检索

### 4.5 行为要求

- 系统从索引中检索与问题最相关的代码片段
- 返回 Top-K 结果（K 为可配置参数）

### 4.5 输出

```json
[
  {
    "chunk_id": "string",
    "content": "string",
    "file_path": "string"
  }
]
```

---

## 4.6 上下文构建

### 4.6 行为要求

- 将多个代码片段组合为统一上下文
- 上下文必须包含：
  - 代码内容
  - 文件路径

- 必须控制上下文大小在合理范围内（避免过长）

---

## 4.7 回答生成

### 4.7 行为要求

- 回答必须基于提供的代码上下文
- 回答必须包含解释性内容
- 不允许凭空编造未在代码中出现的信息

---

## 4.8 返回结果

### 4.8 输出结构

```json
{
  "answer": "string",
  "references": [
    {
      "file_path": "string",
      "snippet": "string"
    }
  ]
}
```

---

## 五、前端功能需求

## 5.1 仓库管理界面

功能：

- 输入代码库路径或地址
- 触发“索引构建”
- 显示当前状态（未加载 / 已加载 / 已索引）

---

## 5.2 问答界面

功能：

- 输入问题
- 展示回答
- 展示引用代码片段

---

## 5.3 交互要求

- 每条回答必须展示对应代码来源
- 用户可以区分回答与代码引用

---

## 六、约束条件（Constraints）

1. 系统必须支持中等规模代码库（不要求超大规模）
2. 每个问题必须在合理时间内返回结果（不可无限等待）
3. 对于问答接口（`/api/ask`）：
   - 若成功返回（`code=0`），`data.answer` 必须为非空字符串，且至少包含一句可读解释
   - 若失败返回（`code!=0`），必须返回明确 `message`，`data` 为 `null`
   - 不允许出现“成功但 answer 为空”的情况
4. 所有引用代码必须真实存在于代码库中

---

## 七、错误处理（Error Handling）

系统需处理以下情况：

统一错误响应约定：

- 所有 API（成功与失败）统一采用 `{ code, message, data }` 结构
- `code=0` 表示成功，`code!=0` 表示失败
- 失败时 `data` 必须为 `null`

## 7.1 无法检索到相关代码

返回：

```json
{
  "code": 3001,
  "message": "未找到相关代码，请尝试更具体的问题",
  "data": {
    "answer": "未找到相关代码，请尝试更具体的问题",
    "references": []
  }
}
```

说明：`3001` 属于业务可预期结果，不视为系统异常；前端应展示提示文案并允许用户继续提问。

---

## 7.2 索引未构建

返回明确错误提示：

```json
{
  "code": 2001,
  "message": "请先构建索引",
  "data": null
}
```

---

## 7.3 代码库加载失败

返回：

```json
{
  "code": 1001,
  "message": "目录不存在、不可读或仓库拉取失败",
  "data": null
}
```

---

## 八、验收标准（Acceptance Criteria）

系统必须满足：

## 功能正确性

1. 成功导入一个代码库
2. 成功构建索引
3. 能回答以下类型问题：
   - 函数作用说明
   - 模块位置查询
   - 基础调用关系

---

## 回答质量

1. 在预置题集（不少于 20 题）上，回答与代码事实一致率 ≥ 80%
2. 每条成功回答至少包含 1 条引用，且引用代码在仓库中可定位
3. 引用中 `file_path` 与 `snippet` 必须可在对应文件中匹配到（允许空白符差异）

---

## 稳定性

1. 不出现崩溃或无响应
2. 错误情况有明确提示

---

## 九、扩展预留（非实现）

系统设计需允许未来扩展：

- 自动代码修改能力
- 更复杂的代码关系分析
- 多轮对话能力

---

## 十、交付物

必须包含：

1. 可运行系统
2. 使用说明
3. 示例测试（至少一个代码库）
